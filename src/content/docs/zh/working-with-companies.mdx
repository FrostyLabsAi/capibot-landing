# 与 AI 公司协作

掌握运行自主 AI 组织的艺术。

## 管理模式深度解析

### 托管模式

**是什么：**
您在执行前批准所有重大决策。

**需要什么批准：**

- ✅ 商业计划
- ✅ Agent 招聘
- ✅ 预算增加
- ✅ 重大计划变更
- ✅ 标记为"人工审查"的任务

**自动发生什么：**

- 批准计划内的任务执行
- 团队协调
- 状态更新
- 文件操作
- 通信

**何时使用：**

- 学习系统
- 重要项目
- 需要控制时
- 新的 AI 公司
- 敏感工作

**示例工作流：**

```
1. 您创建 Atlas 担任 CEO 的公司
2. Atlas 提交商业计划 → **等待批准**
3. 您审查计划 → **批准**
4. Atlas 聘请 Echo → **等待批准**
5. 您批准招聘 → **批准**
6. Echo 完成博客文章 → **人工审查**
7. 您审查文章 → **批准** → 完成
```

### 自主模式

**是什么：**
CEO 在批准的计划内无需询问即可做出决策。

**CEO 可以做什么：**

- ✅ 立即招聘 Agent
- ✅ 无需批准创建任务
- ✅ 执行批准的计划
- ✅ 做出战术决策
- ✅ 协调团队

**仍需要什么批准：**

- 初始商业计划
- 重大预算变更
- 计划修改
- 公司关闭

**何时使用：**

- 日常运营
- 受信任的 CEO Agent
- 经过验证的工作流
- 无需干预的自动化
- 扩展运营

**示例工作流：**

```
1. 您创建 Atlas 担任 CEO 的公司
2. Atlas 提交商业计划 → **批准**
3. Atlas 聘请 Echo → **自动批准**
4. Atlas 聘请 Pixel → **自动批准**
5. Echo 完成博客文章 → **自动完成**
6. Atlas 报告："All blog posts done"
7. 您审查每周摘要
```

### 选择模式

| 因素 | 托管 | 自主 |
|------|------|------|
| **学习曲线** | 较慢、更安全 | 较快、风险较高 |
| **速度** | 取决于您的可用性 | 快速执行 |
| **控制** | 高 | 中 |
| **需要的关注** | 每日 | 每周 |
| **最适合** | 新用户、关键工作 | 有经验的用户、日常运营 |

**混合方法：**
从托管开始，然后切换：

```
第 1-2 周：托管（了解 CEO 风格）
第 3-4 周：在托管中密切监控
第 5 周+：切换到自主
持续：每周检查，需要时干预
```

## 公司生命周期

### 第 1 阶段：简报

**您的工作：**

- 清晰地定义使命
- 设定现实目标
- 选择合适的 CEO
- 设定预算
- 选择模式

**成功技巧：**

- 具体的使命胜过模糊的
- 最多 2-4 个目标
- 将 CEO 与领域匹配
- 从较小的预算开始

### 第 2 阶段：规划

**您的工作：**

- 等待 CEO 提交计划
- 彻底审查
- 提问
- 如需要请求更改
- 准备好时批准

**审查什么：**

- 策略有意义？
- 团队结构合适？
- 时间表现实？
- 预算充足？
- 目标可实现？

### 第 3 阶段：运营

**您的工作（托管）：**

- 审查每日批准
- 监控任务板
- 检查交付物
- 回答问题
- 调整方向

**您的工作（自主）：**

- 每周检查
- 审查摘要
- 监控成本
- 如偏离轨道则干预
- 庆祝胜利

### 第 4 阶段：完成

**您的工作：**

- 审查最终交付物
- 检查所有目标是否实现
- 归档公司
- 保存经验
- 表彰团队

## 何时使用公司与其他选项

### 何时使用公司

✅ **持续运营**

- 每月营销活动
- 持续支持
- 定期内容生产

✅ **复杂项目**

- 需要多个专家
- 顺序工作流
- 需要协调

✅ **一致流程**

- 重复相同工作流
- 标准化输出
- 需要质量控制

✅ **可扩展性**

- 增长的工作量
- 需要增加容量
- 并行执行

### 何时使用个人 Agent

❌ **一次性任务**

- 单个研究问题
- 快速内容创作
- 简单修复

❌ **探索性工作**

- 测试想法
- 原型设计
- 学习

❌ **需要直接控制**

- 手把手指导
- 实时协作
- 教学

❌ **简单范围**

- 只需要一个专业
- 明确的交付物
- 无需协调

### 决策流程图

```
这是持续的还是一次性的？
├── 持续 → 使用公司
└── 一次性 → 对 1 个 Agent 足够简单？
    ├── 是 → 使用个人 Agent
    └── 否 → 需要多个 Agent？
        ├── 是 → 使用公司
        └── 否 → 使用带生成的个人 Agent
```

## 与子 Agent 协作

### 什么是子 Agent？

为特定任务生成的临时 Agent：

- 由父 Agent 创建
- 独立工作
- 回报结果
- 完成后消失

### 何时生成 vs 招聘

| 生成 | 招聘 |
|------|------|
| 并行处理 | 持续的团队成员 |
| 一次性分析 | 核心角色 |
| 激增能力 | 永久职位 |
| 探索性工作 | 明确职责 |
| 示例：5 个研究员分析 5 个竞争对手 | 示例：活动的营销策略师 |

### 示例：为并行工作生成

```
您：Research our top 10 competitors

Scout (Research Analyst):
  "That's a lot of research. I'll spawn
   5 subagents to work in parallel."

Spawned:
  • Subagent 1 → Competitors 1-2
  • Subagent 2 → Competitors 3-4
  • Subagent 3 → Competitors 5-6
  • Subagent 4 → Competitors 7-8
  • Subagent 5 → Competitors 9-10

[30 minutes later]

Scout: "All research complete. Here's the
        consolidated analysis..."
```

### 示例：在公司中生成

```
Q1 Marketing Campaign (Atlas as CEO)

Atlas: "We need market analysis for 5 regions.
       I'll spawn regional researchers."

Spawned within company:
  • Researcher-NorthAmerica
  • Researcher-Europe
  • Researcher-AsiaPacific
  • Researcher-LATAM
  • Researcher-MEA

Each reports to Atlas
Results compiled into master report
Deliverable saved to knowledge base
```

## 常见公司模式

### 模式 1：内容工厂

**结构：**

```
CEO (Content Strategist)
├── Writer 1
├── Writer 2
├── Editor
└── Designer
```

**工作流：**

1. CEO 创建内容日历
2. 撰稿人起草帖子
3. 编辑审查
4. 设计师创建视觉效果
5. 发布到博客/社交

**用于：** 博客、社交媒体、新闻通讯

### 模式 2：产品开发

**结构：**

```
CEO (Product Manager)
├── Designer
├── Frontend Engineer
├── Backend Engineer
└── QA Tester
```

**工作流：**

1. CEO 定义功能
2. 设计师创建模型
3. 工程师构建
4. QA 测试
5. 部署到生产

**用于：** 软件功能、产品迭代

### 模式 3：支持团队

**结构：**

```
CEO (Support Lead)
├── Support Agent 1
├── Support Agent 2
└── Support Agent 3
```

**工作流：**

1. 工单通过集成到达
2. Agent 轮询分配
3. Agent 解决问题
4. 解决方案记录
5. 知识库增长

**用于：** 客户支持、帮助台

### 模式 4：研究团队

**结构：**

```
CEO (Research Lead)
├── Analyst 1
├── Analyst 2
└── Writer
```

**工作流：**

1. CEO 定义研究范围
2. 分析师收集数据
3. 分析师综合发现
4. 撰稿人创建报告
5. 交付物呈现

**用于：** 市场研究、竞争分析

## 扩展公司

### 增加容量

**选项 1：招聘更多 Agent**

- 根据需要添加专家
- 扩展团队能力
- 并行处理

**选项 2：生成子 Agent**

- 临时激增能力
- 并行任务执行
- 无长期承诺

**选项 3：创建子公司**

- 特定功能的子公司
- 独立但相关
- 向母公司汇报

### 管理增长

**需要更多 Agent 的迹象：**

- 任务在收件箱中积压
- Agent 持续忙碌
- 错过截止日期
- 质量下降

**Agent 过多的迹象：**

- Agent 经常空闲
- 协调开销
- 成本太高
- 重复工作

## 公司故障排除

**Company Stuck in Planning：**

- CEO 可能需要更多信息
- 检查 Chat 中的问题
- 提供澄清
- 或修改简报

**Tasks Not Moving：**

- 检查 Agent 可用性
- 验证任务未被阻塞
- 审查依赖项
- 可能需要生成更多 Agent

**Quality Issues：**

- 审查验收标准
- 检查 Agent 能力
- 添加审查员步骤
- 提供示例

**Budget Overruns：**

- 审查成本分解
- 考虑更便宜的模型
- 减少团队规模
- 设置更严格的限制

**Communication Breakdown：**

- 检查公司聊天
- 审查活动流
- 确保 Agent 在线
- 可能需要新 CEO

## 最佳实践

### 设置

1. **Clear Mission** — 一句话目标
2. **Realistic Scope** — 可在时间表内完成
3. **Right CEO** — 领域专业知识
4. **Appropriate Budget** — 意外的缓冲
5. **Measurable Goals** — 知道何时完成

### 运营

1. **Daily Check (Managed)** — 审查批准
2. **Weekly Review (Autonomous)** — 摘要检查
3. **Monitor Costs** — 跟踪支出
4. **Archive When Done** — 保持仪表板整洁
5. **Learn and Iterate** — 下次改进

### 通信

1. **Regular Updates** — 使用公司聊天
2. **Clear Feedback** — 具体且可操作
3. **Celebrate Wins** — 表彰成就
4. **Course Correct** — 偏离轨道时调整
5. **Document Decisions** — 保存到知识库
