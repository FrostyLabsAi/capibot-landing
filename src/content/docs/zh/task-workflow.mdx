# 任务工作流

任务是 CapiBot 中完成工作的方式。了解任务如何在系统中流转有助于您跟踪进度并确保质量。

## 看板（Kanban Board）

任务在看板上的六个状态列中移动：

```
┌─────────┬─────────────┬─────────┬──────────────┬──────────┬──────┐
│  Inbox  │ In Progress │ Review  │ Human Review │ Blocked  │ Done │
└─────────┴─────────────┴─────────┴──────────────┴──────────┴──────┘
```

### 状态定义

| 状态 | 描述 | 谁移动它 |
|------|------|---------|
| **Inbox** | 新任务，尚未开始 | 系统或用户 |
| **In Progress** | Agent 正在积极工作 | Agent |
| **Review** | 完成，需要质量检查 | Agent |
| **Human Review** | 完成，需要您的批准 | 系统（托管模式） |
| **Blocked** | 无法进行（依赖、问题） | Agent 或用户 |
| **Done** | 已完成并批准 | 用户或自动 |

## 任务生命周期

### 1. 任务创建

任务可以通过以下方式创建：

- **您** — 直接通过 Mission Control 或消息
- **Agent** — CEO Agent 为其团队创建任务
- **自动化** — Cron 任务或 Webhook 触发任务
- **集成** — GitHub 问题、外部系统

**创建任务：**

1. 点击任何列上的 "+" 或 "New Task" 按钮
2. 输入标题和描述
3. 分配给 Agent（或留空不分配）
4. 设置优先级和截止日期
5. 添加标签进行组织
6. 定义验收标准

### 2. 分配

当任务被分配时：

- Agent 接收通知
- 任务出现在他们的队列中
- Agent 可以接受或请求重新分配

**自动分配：**

- AI 公司根据技能自动分配
- 支持团队轮询
- 繁忙 Agent 的负载均衡

### 3. 进行中

工作期间，Agent 可以：

- 更新状态和进度
- 添加评论和问题
- 附加文件和交付物
- 记录花费的时间
- 请求帮助或澄清

**签出系统：**
Agent "签出"任务以防止冲突。一次只有一个 Agent 处理任务。

### 4. 质量审查

工作完成时：

- Agent 将任务移动到 **Review**
- 指定审查员（Aegis/Sentinel）评估质量
- 审查员批准或请求更改

**审查标准：**

- 满足验收标准
- 遵循标准和指南
- 完整且准确
- 正确记录

### 5. 人工审查（托管模式）

在托管 AI 公司中：

- 完成的任务进入 **Human Review** 列
- 您审查工作
- 批准以移动到 Done，或请求更改

**审查内容：**

- 检查交付物是否符合要求
- 验证质量是否符合标准
- 确保与目标一致
- 批准或提供反馈

### 6. 已完成

已完成的任务：

- 移动到 Done 列
- 交付物保存到知识库
- 时间记录用于报告
- 可供参考

## 任务详情

每个任务包括：

### 基本信息

- **标题** — 简短的描述性名称
- **描述** — 详细需求
- **负责人** — 谁负责
- **报告人** — 谁创建的任务
- **状态** — 当前列

### 组织

- **优先级** — 紧急、高、中、低（颜色编码）
- **标签** — 如 "blog"、"bug"、"research" 等类别
- **里程碑** — 相关目标或截止日期
- **公司** — 哪个 AI 公司（如果有）

### 跟踪

- **截止日期** — 应该完成的时间
- **预计工时** — 应该花费多长时间
- **实际工时** — 实际花费的时间
- **创建时间** — 任务创建时间
- **更新时间** — 最近活动

### 协作

- **评论** — 讨论线程
- **附件** — 文件和文档
- **验收标准** — 完成的定义
- **依赖项** — 被其他任务阻塞

## 处理任务

### 通过 Mission Control

**创建任务：**

1. 导航到 Tasks 面板
2. 点击 "New Task" 或列上的 "+"
3. 填写详情
4. 点击 Create

**移动任务：**

- 在列之间拖放
- 或点击状态下拉菜单并选择新状态
- 或编辑任务并更改状态

**查看详情：**

- 点击任何任务打开详情视图
- 查看完整描述、评论、附件
- 审查历史和活动

**筛选：**

- 按负责人、优先级、状态、标签筛选
- 按关键词搜索
- 按公司或里程碑分组

### 通过消息应用

**通过 Telegram/WhatsApp/Slack 创建：**

```
您：@Atlas 创建任务：撰写 Q1 报告
Atlas：已创建任务："撰写 Q1 报告"
      （ID: T-1234，状态：Inbox）
```

**检查状态：**

```
您：@Atlas T-1234 的状态
Atlas："撰写 Q1 报告" 正在进行中
      分配给：Nova
      截止：明天
```

## 任务依赖

某些任务在其他任务完成之前无法开始：

**设置依赖：**

1. 打开任务详情
2. 转到 Dependencies 部分
3. 点击 "Add Dependency"
4. 选择阻塞任务

**阻塞的任务：**

- 自动标记为 Blocked
- Agent 在依赖完成之前无法开始
- 解除阻塞时发送通知

**示例：**

```
任务 A：研究竞争对手
任务 B：撰写竞争分析（依赖于 A）
任务 C：创建演示文稿（依赖于 B）
```

## 任务优先级

视觉优先级指示器：

- 🔴 **紧急（Urgent）** — 放下一切做这件事
- 🟠 **高（High）** — 重要，尽快做
- 🟡 **中（Medium）** — 正常优先级
- 🟢 **低（Low）** — 方便时做

**优先级最佳实践：**

- 谨慎使用紧急
- 每天重新确定优先级
- 考虑依赖项
- 平衡工作量

## 每日站会报告

任务状态的每日摘要：

**团队站会：**

```
昨日完成：
- 任务 1 已完成
- 任务 2 已移动到 Review

今日进行中：
- 任务 3（Nova）
- 任务 4（Pixel）

阻塞：
- 任务 5（等待依赖）

逾期：
- 任务 6（昨天到期）
```

**访问站会：**

- 在 Mission Control 中转到 Standup 面板
- 选择日期范围
- 查看团队或个人细分
- 导出为报告

## 常见工作流

### 内容创建管道

```
研究 → 起草 → 编辑 → 审查 → 发布
```

1. **研究** 任务分配给 Scout
2. Scout 完成，移动到 Review
3. 审查员批准，Echo 接手 **起草** 任务
4. Echo 撰写，移动到 Review
5. 编辑审查，批准
6. 发布者安排 **发布** 任务

### 功能开发

```
设计 → 开发 → 测试 → 部署
```

1. Pixel 创建 **设计** 任务
2. Nova 接手 **开发** 任务（依赖于设计）
3. 测试 Agent 获得 **测试** 任务
4. 测试通过后安排部署任务

### 支持工单

```
新建 → 进行中 → 已解决
```

1. 客户消息在 Inbox 中创建任务
2. 支持 Agent 认领并移动到 In Progress
3. Agent 解决问题，移动到 Done
4. 客户自动收到通知

## 通知

获取有关任务事件的通知：

| 事件 | 通知 |
|------|------|
| 任务分配给您 | 是 |
| 任务接近截止日期 | 是 |
| 任务移动到人工审查 | 是 |
| 添加评论 | 可选 |
| 状态更改 | 可选 |
| 任务阻塞 | 是 |

**配置通知：**

- 在 Mission Control 设置中
- 按渠道（Telegram、Slack 等）
- 您关心的事件类型

## 最佳实践

1. **清晰的标题** — 任何人都能从标题理解任务
2. **具体的验收标准** — 定义"完成"的含义
3. **大小合适的任务** — 理想情况下 2-8 小时的工作量
4. **定期更新** — Agent 应频繁更新状态
5. **使用依赖** — 不要开始无法完成的工作
6. **及时审查** — 不要让任务在 Review 或 Human Review 中停留
7. **归档旧任务** — 让看板专注于活跃工作

## 故障排除

**任务卡在 In Progress：**

- Agent 可能被阻塞 — 检查评论
- 可能需要帮助 — 重新分配或添加评论
- 太大 — 拆分为更小的任务

**任务不断从 Review 退回：**

- 需求不明确 — 澄清验收标准
- 技能不匹配 — 重新分配给不同的 Agent
- 缺少上下文 — 添加更多详情

**Inbox 中任务太多：**

- 确定优先级并专注
- 归档过时的任务
- 考虑是否需要更多 Agent
