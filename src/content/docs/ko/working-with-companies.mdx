# AI 기업과 함께 작업하기

자율 AI 조직을 운영하는 기술을 마스터하세요.

## 관리 모드 심층 분석

### 관리형 모드

**무엇인가:**
모든 중요한 결정이 실행되기 전에 승인합니다.

**승인이 필요한 것:**

- ✅ 비즈니스 계획
- ✅ 에이전트 채용
- ✅ 예산 증가
- ✅ 주요 계획 변경
- ✅ "Human Review"로 표시된 작업

**자동으로 발생하는 것:**

- 승인된 계획 내 작업 실행
- 팀 조정
- 상태 업데이트
- 파일 작업
- 커뮤니케이션

**사용 시기:**

- 시스템 학습
- 중요한 프로젝트
- 통제를 원할 때
- 새로운 AI 기업
- 민감한 작업

**예시 워크플로우:**

```
1. Atlas를 CEO로 기업 생성
2. Atlas가 비즈니스 계획 제출 → **승인 대기 중**
3. 계획 검토 → **승인**
4. Atlas가 Echo 고용 → **승인 대기 중**
5. 고용 승인 → **승인**
6. Echo가 블로그 포스트 완료 → **Human Review**
7. 포스트 검토 → **승인** → 완료됨
```

### 자율 모드

**무엇인가:**
CEO가 승인된 계획 내에서 묻지 않고 결정을 내립니다.

**CEO가 할 수 있는 것:**

- ✅ 즉시 에이전트 고용
- ✅ 승인 없이 작업 생성
- ✅ 승인된 계획 실행
- ✅ 전술적 결정
- ✅ 팀 조정

**여전히 승인이 필요한 것:**

- 초기 비즈니스 계획
- 주요 예산 변경
- 계획 수정
- 기업 종료

**사용 시기:**

- 일상적인 운영
- 신뢰할 수 있는 CEO 에이전트
- 입증된 워크플로우
- 핸즈오프 자동화
- 확장 작업

**예시 워크플로우:**

```
1. Atlas를 CEO로 기업 생성
2. Atlas가 비즈니스 계획 제출 → **승인**
3. Atlas가 Echo 고용 → **자동 승인**
4. Atlas가 Pixel 고용 → **자동 승인**
5. Echo가 블로그 포스트 완료 → **자동 완료**
6. Atlas 보고: "모든 블로그 포스트 완료"
7. 주간 요약 검토
```

### 모드 선택

| 요인               | 관리형                      | 자율                         |
| ------------------ | --------------------------- | ------------------------------ |
| **학습 곡선**      | 느리지만 안전               | 빠르지만 위험                |
| **속도**           | 가용성에 따라 다름          | 빠른 실행                    |
| **통제**           | 높음                        | 중간                         |
| **필요한 주의**    | 매일                        | 주간                         |
| **가장 적합한 경우** | 새로운 사용자, 중요한 작업  | 경험 있는 사용자, 일상 운영  |

**하이브리드 접근:**
관리형으로 시작한 다음 전환:

```
1-2주: 관리형 (CEO의 스타일 학습)
3-4주: 관리형에서 면접히 모니터링
5주+: 자율로 전환
진행 중: 주간 확인, 필요시 개입
```

## 기업 라이프사이클

### 1단계: 브리핑

**사용자의 작업:**

- 미션을 명확하게 정의
- 현실적인 목표 설정
- 적합한 CEO 선택
- 예산 설정
- 모드 선택

**성공 팁:**

- 구체적인 미션이 모호한 것보다 낫습니다
- 최대 2-4개 목표
- 도메인에 맞는 CEO 매칭
- 더 작은 예산으로 시작

### 2단계: 계획

**사용자의 작업:**

- CEO가 계획을 제출할 때까지 기다리기
- 철저히 검토
- 질문하기
- 필요한 경우 변경 요청
- 준비되면 승인

**검토할 것:**

- 전략이 합리적인가?
- 팀 구조가 적절한가?
- 타임라인이 현실적인가?
- 예산이 적절한가?
- 목표가 달성 가능한가?

### 3단계: 운영

**사용자의 작업 (관리형):**

- 매일 승인 검토
- 작업 보드 모니터링
- 산출물 확인
- 질문에 답변
- 방향 수정

**사용자의 작업 (자율):**

- 주간 체크인
- 요약 검토
- 비용 모니터링
- 궤도를 벗어나면 개입
- 성과 축하

### 4단계: 완료

**사용자의 작업:**

- 최종 산출물 검토
- 모든 목표가 달성되었는지 확인
- 기업 보관
- 교훈 저장
- 팀 인정

## 기업 대 다른 옵션을 언제 사용할 것인가

### 기업을 사용할 때:

✅ **지속적인 운영**

- 월간 마케팅 캠페인
- 지속적인 지원
- 정기적인 콘텐츠 제작

✅ **복잡한 프로젝트**

- 여러 전문가 필요
- 순차 워크플로우
- 조정 필요

✅ **일관된 프로세스**

- 반복되는 동일한 워크플로우
- 표준화된 출력
- 품질 관리 필요

✅ **확장성**

- 증가하는 워크로드
- 용량 추가 필요
- 세부 관리 없이 확장

### 개별 에이전트를 사용할 때:

❌ **일회성 작업**

- 단일 연구 질문
- 빠른 콘텐츠 조각
- 간단한 수정

❌ **탐색적 작업**

- 아이디어 테스트
- 프로토타이핑
- 학습

❌ **직접 통제 필요**

- 실습 지침
- 실시간 협업
- 교육

❌ **간단한 범위**

- 하나의 전문성 필요
- 명확한 산출물
- 조정 없음

### 결정 순서도

```
지속적인가 아니면 일회성인가?
├── 지속적인 → 기업 사용
└── 일회성 → 1명의 에이전트로 충분히 간단한가?
    ├── 예 → 개별 에이전트 사용
    └── 아니오 → 여러 에이전트가 필요한가?
        ├── 예 → 기업 사용
        └── 아니오 → 생성과 함께 개별 에이전트 사용
```

## 서브에이전트와 함께 작업하기

### 서브에이전트란 무엇인가?

특정 작업을 위한 임시 에이전트:

- 상위 에이전트에 의해 생성
- 독립적으로 작업
- 결과 보고
- 완료되면 사라짐

### 생성 대 채용

| 생성                                          | 채용                                       |
| --------------------------------------------- | ------------------------------------------ |
| 병렬 처리                                     | 지속적인 팀원                              |
| 일회성 분석                                   | 핵심 역할                                  |
| 과부하 용량                                   | 영구 직책                                  |
| 탐색적 작업                                   | 정의된 책임                                |
| 예시: 5개 경쟁사를 분석하는 5명의 연구원 생성 | 예시: 캠페인을 위한 마케팅 전략가 고용     |

### 예시: 병렬 작업을 위한 생성

```
사용자: 상위 10개 경쟁사를 조사해줘

Scout (연구 분석가):
  "많은 연구가 필요합니다. 병렬로
   작업할 5개의 서브에이전트를 생성하겠습니다."

생성됨:
  • 서브에이전트 1 → 경쟁사 1-2
  • 서브에이전트 2 → 경쟁사 3-4
  • 서브에이전트 3 → 경쟁사 5-6
  • 서브에이전트 4 → 경쟁사 7-8
  • 서브에이전트 5 → 경쟁사 9-10

[30분 후]

Scout: "모든 연구 완료. 통합 분석은
        다음과 같습니다..."
```

### 예시: 기업에서 생성

```
Q1 마케팅 캠페인 (Atlas를 CEO로)

Atlas: "5개 지역에 대한 시장 분석이 필요합니다.
       지역 연구원을 생성하겠습니다."

기업 내에서 생성:
  • Researcher-NorthAmerica
  • Researcher-Europe
  • Researcher-AsiaPacific
  • Researcher-LATAM
  • Researcher-MEA

각각 Atlas에게 보고
결과가 마스터 보고서로 컴파일됨
산출물이 지식 베이스에 저장됨
```

## 일반적인 기업 패턴

### 패턴 1: 콘텐츠 팩토리

**구조:**

```
CEO (콘텐츠 전략가)
├── 작가 1
├── 작가 2
├── 편집자
└── 디자이너
```

**워크플로우:**

1. CEO가 콘텐츠 달력 생성
2. 작가들이 포스트 초안 작성
3. 편집자가 검토
4. 디자이너가 시각 자산 생성
5. 블로그/소셜에 게시

**사용:** 블로그, 소셜 미디어, 뉴스레터

### 패턴 2: 제품 개발

**구조:**

```
CEO (제품 관리자)
├── 디자이너
├── 프론트엔드 엔지니어
├── 백엔드 엔지니어
└── QA 테스터
```

**워크플로우:**

1. CEO가 기능 정의
2. 디자이너가 목업 생성
3. 엔지니어가 구축
4. QA가 테스트
5. 프로덕션에 배포

**사용:** 소프트웨어 기능, 제품 반복

### 패턴 3: 지원 팀

**구조:**

```
CEO (지원 리드)
├── 지원 에이전트 1
├── 지원 에이전트 2
└── 지원 에이전트 3
```

**워크플로우:**

1. 통합을 통해 티켓 도착
2. 에이전트가 라운드 로빈으로 할당
3. 에이전트가 문제 해결
4. 해결책이 문서화됨
5. 지식 베이스 성장

**사용:** 고객 지원, 헬프 데스크

### 패턴 4: 연구 팀

**구조:**

```
CEO (연구 리드)
├── 분석가 1
├── 분석가 2
└── 작가
```

**워크플로우:**

1. CEO가 연구 범위 정의
2. 분석가들이 데이터 수집
3. 분석가들이 발견 종합
4. 작가가 보고서 생성
5. 산출물 제시

**사용:** 시장 조사, 경쟁 분석

## 기업 확장

### 용량 추가

**옵션 1: 더 많은 에이전트 고용**

- 필요에 따라 전문가 추가
- 팀 능력 확장
- 병렬 처리

**옵션 2: 서브에이전트 생성**

- 임시 과부하 용량
- 병렬 작업 실행
- 장기적인 약속 없음

**옵션 3: 하위 기업 생성**

- 특정 기능을 위한 자회사
- 독립적이지만 관련
- 부모에게 보고

### 성장 관리

**더 많은 에이전트가 필요한 징후:**

- 작업이 Inbox에 쌓임
- 에이전트가 끊임없이 바쁨
- 놓친 마감일
- 품질 저하

**너무 많은 징후:**

- 에이전트가 자주 유휴
- 조정 오버헤드
- 비용이 너무 높음
- 중복된 노력

## 기업 문제 해결

**계획에서 기업이 멈춤:**

- CEO가 더 많은 정보가 필요할 수 있음
- 질문을 위해 채팅 탭 확인
- 명확화 제공
- 또는 브리핑 수정 및 다시 제출

**작업이 이동하지 않음:**

- 에이전트 가용성 확인
- 작업이 차단되지 않았는지 확인
- 의존성 검토
- 더 많은 에이전트를 생성해야 할 수 있음

**품질 문제:**

- 수락 기준 검토
- 에이전트 능력 확인
- 검토자 단계 추가
- 예시 제공

**예산 초과:**

- 비용 분석 검토
- 더 저렴한 모델 고려
- 팀 규모 줄이기
- 더 엄격한 한계 설정

**커뮤니케이션 중단:**

- 기업 채팅 확인
- 활동 피드 검토
- 에이전트가 온라인인지 확인
- 새로운 CEO가 필요할 수 있음

## 모범 사례

### 설정

1. **명확한 미션** — 한 문장 목표
2. **현실적인 범위** — 타임라인 내에서 완료 가능
3. **적합한 CEO** — 도메인 전문성
4. **적절한 예산** — 예상치 못한 것에 대한 버퍼
5. **측정 가능한 목표** — 완료 시점 알기

### 운영

1. **일일 체크 (관리형)** — 승인 검토
2. **주간 검토 (자율)** — 요약 확인
3. **비용 모니터링** — 지출 추적
4. **완료 시 보관** — 대시보드를 깔끔하게 유지
5. **학습 및 반복** — 다음 번에 개선

### 커뮤니케이션

1. **정기적인 업데이트** — 기업 채팅 사용
2. **명확한 피드백** — 구체적이고 실행 가능
3. **성과 축하** — 성취 인정
4. **방향 수정** — 궤도를 벗어나면 조정
5. **결정 문서화** — 지식 베이스에 저장
