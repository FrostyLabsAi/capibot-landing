## LLM 제공자 설정

사용할 AI 모델을 구성합니다.

### 제공자 우선순위

```
LLM 제공자 우선순위

CapiBot에 AI가 필요할 때 이 순서로 제공자를 시도:

1. 🥇 Anthropic Claude
2. 🥈 Google Gemini
3. 🥉 Kimi (Moonshot)
4.    GLM (Zhipu AI)
5.    MiniMax
6.    Ollama (로컬)

[제공자 추가] [재정렬]
```

**장애 조치 작동 방식:**

1. 기본 제공자 (Claude)가 요청 시도
2. 실패하면 자동으로 다음 구성된 제공자 시도
3. 성공하거나 모두 실패할 때까지 계속

### 제공자 구성

**Anthropic:**

```
API Key: [sk-ant-********************************    ]
모델: [Claude 3.5 Sonnet ▼]
최대 토큰: 4096
온도: 0.7
[연결 테스트]
```

**Ollama (로컬):**

```
Base URL: [http://localhost:11434           ]
모델: [llama3.1:latest ▼]
[모델 새로고침]
```

**Google Gemini:**

```
API Key: [************************************    ]
모델: [Gemini 1.5 Pro ▼]
[키 확인]
```

### 모델 재정의

에이전트별 모델 선택:

```

Scout (연구) → Claude (더 나은 분석)
```

## 데이터 보존

데이터를 얼마나 오래 보관할지 제어.

```
데이터 보존 정책

자동 정리가 오래된 데이터를 자동으로 제거합니다.

활동        [30 ▼]일 동안 보관
알림        [7 ▼]일 동안 보관
감사 로그   [90 ▼]일 동안 보관
통신/메시지 [15 ▼]일 동안 보관
웹훅 내역   [30 ▼]일 동안 보관
추적        [30 ▼]일 동안 보관
파일        [90 ▼]일 동안 보관
완료된 작업 [365 ▼]일 동안 보관

☑️ 자동 정리 활성화
☐ 삭제 전 보관
[변경 사항 저장]
```

**자동 정리를 하는 이유:**

- 데이터베이스 크기 감소
- 성능 향상
- 개인 정보 보호 유지
- 비용 제어

**중요한 데이터는 절대 삭제되지 않음:**

- 활성 작업
- 에이전트 메모리
- 지식 베이스
- 기업 정보
- 설정

## 모범 사례

### 구성

1. **변경 사항 문서화** — 설정을 변경한 이유 메모
2. **먼저 테스트** — 개발 환경에서 변경 시도
3. **먼저 백업** — 주요 변경 전에
4. **보수적으로 시작** — 점진적으로 강화
5. **정기적으로 검토** — 월간 설정 감사

### 보안

1. **강력한 API 키** — 자동 생성된 키 사용
2. **정기적인 순환** — 90일마다
3. **최소 권한** — 필요한 최소한의 접근
4. **로그 모니터링** — 이상 활동 감시
5. **업데이트 유지** — 최신 버전의 보안 패치

### 성능

1. **적절한 풀 크기** — 사용량에 맞는 DB 연결
2. **현명하게 캐싱** — 동적 콘텐츠는 캐싱하지 않기
3. **정기적으로 정리** — 오래된 데이터 자동 정리
4. **리소스 모니터링** — 메모리와 디스크 주시
5. **쿼리 최적화** — 느린 쿼리 로그 확인

## 문제 해결

**설정이 저장되지 않음:**

- 권한 확인
- 값이 유효한지 확인
- 오류 메시지 검토
- 페이지 새로고침 시도

**연결 테스트 실패:**

- API 키 확인
- 네트워크 연결 확인
- 방화벽 규칙 검토
- 서비스 상태 확인

**성능 문제:**

- 연결 풀 크기 낮추기
- 캐시 기간 줄이기
- 자동 정리 활성화
- 데이터베이스 크기 확인

**보안 경고:**

- 감사 로그 검토
- 접근 패턴 확인
- 화이트리스트 확인
- 손상된 키 순환
