# 스킬

스킬은 코드를 작성하지 않고도 CapiBot 에이전트에게 새로운 기능을 가르치는 방법입니다. 스킬은 에이전트가 특정 작업을 수행하는 방법이나 특정 도구로 작업하는 방법을 배울 수 있는 전문화된 교육 매뉴얼이라고 생각하세요.

## 스킬이란 무엇인가요?

스킬은 에이전트에게 특정 작업을 수행하거나 특정 도구로 작업하는 방법을 가르치는 **지침이 있는 Markdown 파일**입니다. 에이전트가 무언가를 해야 할 때 관련 스킬을 읽고 지침을 따릅니다.

```
┌─────────────────────────────────────────────────────────┐
│  사용자 요청: "일일 보고서 예약"                          │
│                                                         │
│  에이전트: "이걸 위해 cron 스킬이 필요해"                │
│                                                         │
│  ┌──────────────┐                                       │
│  │  스킬 읽기   │                                       │
│  │  "cron"      │                                       │
│  └──────────────┘                                       │
│         ↓                                               │
│  ┌──────────────┐                                       │
│  │  지침 따르기 │                                       │
│  └──────────────┘                                       │
│         ↓                                               │
│  ✓ 작업 완료                                              │
└─────────────────────────────────────────────────────────┘
```

## 내장 스킬

CapiBot은 즉시 사용할 수 있는 8개의 내장 스킬과 함께 제공됩니다:

### 1. Composio — 서드파티 앱 통합

**하는 일:** 1000개 이상의 앱에 연결 (Notion, Linear, Jira, Slack, GitHub 등)

**사용 시기:**

- Notion에서 페이지 생성
- Linear/Jira에서 이슈 생성
- Slack 메시지 본문
- 모든 앱 통합

**예시:**

```
사용자: 회의 노트로 Notion 페이지를 만들어줘

에이전트: [composio 스킬 사용]
       1. 연결된 앱 확인
       2. "페이지 생성" 작업 찾기
       3. 회의 노트로 실행

       ✅ Notion에 "회의 노트" 페이지 생성
```

**핵심 지침:**

- 항상 먼저 연결된 앱 확인
- 실행 전 사용 가능한 작업 나열
- 정확한 작업 슬러그 사용 (예: `NOTION_CREATE_NOTION_PAGE`)
- 쓰기 작업 전 확인

### 2. Cron — 작업 예약

**하는 일:** 미리 알림과 반복 작업 예약

**사용 시기:**

- 일일 스탠드업 보고서
- 주간 요약
- 마감일 미리 알림
- 반복 상태 확인

**예시:**

```
사용자: 매일 오전 9시에 블로그 포스트 검토를
      미리 알려줘

에이전트: [cron 스킬 사용]
       오전 9:00에 매일 미리 알림 예약
       메시지: "블로그 포스트 검토"

       ✅ 매일 미리 알림 설정
```

**시간 표현:**

- "20분마다" → `every_seconds: 1200`
- "매시간" → `every_seconds: 3600`
- "매일 오전 8시" → `cron_expr: "0 8 * * *"`
- "평일 오후 5시" → `cron_expr: "0 17 * * 1-5"`

### 3. GitHub — 저장소 작업

**하는 일:** `gh` CLI를 사용하여 GitHub 저장소 작업

**사용 시기:**

- 이슈 생성
- 풀 리퀘스트 검토
- 저장소 상태 확인
- 릴리스 관리

**예시:**

```
사용자: capibot 저장소에 열린 PR이 있는지 확인해줘

에이전트: [github 스킬 사용]
       열린 풀 리퀘스트 3개:
       1. #45 - 인증 버그 수정
       2. #46 - 다크 모드 지원 추가
       3. #47 - 문서 업데이트
```

### 4. Weather — 날씨 정보

**하는 일:** 현재 날씨와 예보 얻기

**사용 시기:**

- 야외 이벤트 계획
- 여행 준비
- 빠른 날씨 확인

**예시:**

```
사용자: 오늘 보고타 날씨가 어때?

에이전트: [weather 스킬 사용]
       보고타, 콜롬비아
       ☁️  부분적으로 흐림
       🌡️  18°C / 64°F
       💧  65% 습도
       🌬️  산들바람
```

### 5. Summarize — 텍스트 요약

**하는 일:** 긴 콘텐츠 요약

**사용 시기:**

- 웹 기사 요약
- 긴 문서 압축
- YouTube 비디오 요약
- 회의 노트

**예시:**

```
사용자: 이 기사 요약해줘: [긴 URL]

에이전트: [summarize 스킬 사용]
       요약 (3분 읽기):

       • 핵심 포인트 1...
       • 핵심 포인트 2...
       • 핵심 포인트 3...
```

### 6. TMux — 터미널 세션 관리

**하는 일:** tmux 세션 원격 제어

**사용 시기:**

- 오래 실행되는 프로세스 관리
- 원격 서버 관리
- 세션 지속성

**예시:**

```
사용자: 서버 로그를 위한 tmux 세션 시작해줘

에이전트: [tmux 스킬 사용]
       "server-logs" tmux 세션 생성
       실행 중: tail -f /var/log/server.log

       재연결: tmux attach -t server-logs
```

### 7. Skill Creator — 새 스킬 생성

**하는 일:** 전문화된 작업을 위한 커스텀 스킬 생성

**사용 시기:**

- 고유한 워크플로우
- 기업별 프로세스
- 도메인 전문성
- 반복적인 복잡한 작업

**예시:**

```
사용자: 비용 보고서 처리를 위한 스킬을 만들어줘

에이전트: [skill-creator 스킬 사용]
       "expense-processor" 스킬을 만들겠습니다.

       이 스킬은 무엇을 해야 하나요?
       1. 영수증에서 데이터 추출
       2. 비용 분류
       3. 보고서 생성
       4. 회계 시스템으로 납품
```

### 8. Playbook — 전략적 조정

**하는 일:** AI 기업을 위한 다단계 프로젝트 조정

**사용 시기:**

- 스타트업 MVP 개발
- 마케팅 캠페인
- 엔터프라이즈 기능 개발
- 인시던트 대응

**포함:**

- 6단계 라이프사이클 (Discovery, Strategy, Foundation, Build, Hardening, Launch, Operate)
- 시나리오 런북
- 에이전트 조정 템플릿
- 품질 게이트

## 스킬 작동 방식

### 점진적 공개

스킬은 효율성을 위해 3단계 시스템을 사용합니다:

**레벨 1: 메타데이터 (항상 컨텍스트에 있음)**

```yaml
---
name: composio
description: "Composio를 통해 1000개 이상의 서드파티 앱 사용..."
---
```

- ~100단어
- 에이전트가 스킬을 사용할 시기 결정

**레벨 2: SKILL.md 본문 (트리거될 때)**

```markdown
# Composio — 서드파티 앱 통합

## 워크플로우

1. 연결된 앱 확인
2. 사용 가능한 작업 나열
3. 작업 스키마 얻기
4. 작업 실행
```

- 스킬이 필요할 때만 로드
- 실제 지침 포함

**레벨 3: 번들 리소스 (필요에 따라)**

```
skill-folder/
├── SKILL.md (메인 지침)
├── scripts/ (헬퍼 스크립트)
├── references/ (상세 문서)
└── assets/ (템플릿 등)
```

- 참조될 때만 로드
- 메인 스킬 파일을 간결하게 유지

### 스킬 트리거

요청을 하면 에이전트가:

1. **스킬 메타데이터 읽기** — 설치된 모든 스킬
2. **관련 스킬 선택** — 설명 일치에 기반
3. **스킬 지침 로드** — SKILL.md 콘텐츠
4. **지침 따르기** — 워크플로우 실행
5. **번들 리소스 사용** — 필요한 경우

```
사용자: "Notion 페이지 생성"

에이전트 생각:
├─ 스킬 메타데이터 확인
│  ├─ "cron" — 예약, 관련 없음
│  ├─ "composio" — 서드파티 앱 ✓ 일치
│  ├─ "github" — 저장소, 관련 없음
│  └─ "weather" — 날씨, 관련 없음
│
├─ composio 스킬 로드
│  └─ SKILL.md 지침 읽기
│
├─ 워크플로우 실행
│  ├─ 연결된 앱 확인
│  ├─ "페이지 생성" 작업 찾기
│  ├─ 매개변수 스키마 얻기
│  └─ 작업 실행
│
└─ 완료! Notion 페이지 생성
```

### 새 스킬 설치

**커뮤니티 레지스트리에서:**

1. **"레지스트리 탐색"** 클릭
2. 검색 또는 카테고리 탐색
3. 스킬 클릭하여 세부 정보 보기
4. **"Install"** 클릭
5. 즉시 사용 가능

**파일에서:**

1. **"+ 파일에서 설치"** 클릭
2. `.skill` 파일 업로드
3. 설치 확인
4. 즉시 사용 가능

### 스킬 테스트

새 스킬에 의존하기 전에 테스트:

```
스킬: composio
테스트: Notion에 테스트 페이지 생성

✅ 테스트 통과
   Notion에 "테스트" 페이지 생성

   스킬이 올바르게 작동합니다.
```

### 스킬 제거

**스킬 제거:**

1. 스킬 패널에서 스킬 찾기
2. **"..."** 메뉴 클릭
3. **"Uninstall"** 선택
4. 제거 확인

**참고:** 내장 스킬은 제거할 수 없습니다.

## 커스텀 스킬 생성

### 스킬을 만들 시기

다음 경우 커스텀 스킬 생성:

✅ **고유한 워크플로우**

- 기업별 프로세스
- 독점 방법론
- 커스텀 도구 통합

✅ **반복적인 복잡한 작업**

- 자주 수행
- 다단계 프로세스
- 도메인 지식 필요

✅ **전문화된 지식**

- 산업 전문성
- 기업 표준
- 모범 사례

✅ **통합 필요**

- 커스텀 API
- 내부 도구
- 독점 시스템

### 예시: 비용 처리 스킬 생성

**시나리오:** 팀이 특정 단계로 매주 비용 보고서를 처리합니다.

**1단계: 스킬 정의**

```
이름: expense-processor
설명: "회사 정책에 따라 비용 보고서를 처리.
영수증에서 데이터 추출, 비용 분류, 정책 한도에 대해
검증하고, 환불 보고서 생성."
```

**2단계: SKILL.md 작성**

```markdown
---
name: expense-processor
description: "비용 보고서 처리..."
---

# 비용 보고서 처리기

## 워크플로우

1. **데이터 추출**
   - 영수증 이미지 읽기
   - 추출: 날짜, 상점, 금액, 카테고리
   - 합계 검증

2. **분류**
   - 식사: $75 한도
   - 여행: 한도 없음
   - 물품: $100 한도
   - 접대: $50 한도

3. **정책 검증**
   - 카테고리별 한도 확인
   - 예외 플래그
   - 필요한 승인 메모

4. **보고서 생성**
   - 요약 생성
   - CSV로 납품
   - 회계에 제출

## 카테고리 및 한도

| 카테고리 | 한도    | 승인 필요 |
| --------- | ------- | --------- |
| 식사      | $75/식사| >$75     |
| 여행      | 한도 없음| -        |
| 물품      | $100    | >$100    |
| 접대      | $50     | >$50     |
```

**3단계: 스킬 테스트**

```
사용자: 이 비용 보고서 처리 [영수증]

에이전트: [expense-processor 스킬 사용]

       5개 영수증 처리 중...

       ✅ 모든 정책 한도 내

       요약:
       • 식사: $67, $45 (2식)
       • 여행: $340 (항공편)
       • 물품: $23

       합계: $475
       보고서 생성: expense-report-week1.csv
```

## 스킬 모범 사례

### 스킬 사용자의 경우

1. **스킬 알기** — 설치된 것 검토
2. **새 스킬 테스트** — 의존하기 전에
3. **피드백 제공** — 스킬 개선 도움
4. **업데이트 유지** — 최신 버전 설치

### 스킬 작성자의 경우

1. **명확한 설명** — 에이전트가 올바르게 트리거하도록 도움
2. **간결한 지침** — 500줄 이하 유지
3. **점진적 공개** — 세부 정보는 참조에 사용
4. **철저히 테스트** — 실제 시나리오
5. **버전 관리** — 변경 추적

### 훌륭한 스킬의 해부학

**좋은 SKILL.md 구조:**

```markdown
---
name: my-skill
description: "이 스킬이 무엇을 하는지와
사용 시기에 대한 명확하고 포괄적인 설명."
---

# 내 스킬

## 빠른 시작

간략한 개요와 간단한 예시

## 워크플로우

단계별 지침

## 예시

일반적인 사용 사례

## 고급

세부 정보에 대한 참조 링크
```

**나쁜 SKILL.md 구조:**

```markdown
---

# 내 스킬

이론 설명하는 많은 텍스트... ← 너무 장황함!

더 많은 배경 정보... ← 실행 가능하지 않음!
```

## 문제 해결

**스킬이 트리거되지 않음:**

- 설명이 명확하고 구체적인지 확인
- 스킬이 설치되었는지 확인
- 더 구체적인 요청 시도
- 이름으로 스킬을 언급해야 할 수 있음

**스킬이 잘못 트리거됨:**

- 설명이 너무 광범위할 수 있음
- 더 구체적인 트리거 추가
- 여러 스킬로 분할 고려

**스킬 지침이 불분명:**

- 직접 SKILL.md 읽기
- 더 명확한 단계로 업데이트
- 더 많은 예시 추가
- 다양한 입력으로 테스트

**스킬이 오래됨:**

- 업데이트 확인
- 최신 문서 검토
- 번들 리소스 업데이트
- 새 버전 생성 고려
