# 에이전트 계층

CapiBot은 에이전트를 3계층 계층 구조로 구성합니다. 이러한 계층을 이해하면 작업에 적합한 에이전트를 선택하고 효과적인 팀을 구축하는 데 도움이 됩니다.

## 세 계층

### 리드 (CEO/임원)

**광범위한 권한을 가진 전략적 리더**

리드 에이전트는 임원입니다. 다음을 할 수 있습니다:

- 비즈니스 계획과 전략 생성
- 다른 에이전트를 고용하여 팀 구축
- 조직 전체에 작업 위임
- 모든 도구와 커뮤니케이션 채널 접근
- 고위급 결정

**예시:**

- **Atlas** — 비즈니스 계획을 작성하고 운영을 오케스트레이션하는 CEO
- **Helios** — 기술 이니셔티브를 이끄는 CTO
- **Aura** — 마케팅 전략을 감독하는 CMO

**사용 시기:** AI 기업 운영, 전략 계획, 교차 기능 조정

### 전문가

**깊은 기술을 가진 도메인 전문가**

전문가 에이전트는 전문 팀원입니다. 다음을 수행합니다:

- 특정 영역에 대한 깊은 전문성 보유
- 복잡한 작업을 독립적으로 실행
- 병렬 작업을 위해 서브에이전트 생성 가능
- 리드 에이전트에게 보고
- 도메인 내 고급 도구 사용

**예시:**

- **Nova** — 풀스택 엔지니어 (엔드투엔드 기능 구축)
- **Pixel** — UI/UX 디자이너 (인터페이스와 프로토타입 생성)
- **Echo** — 콘텐츠 전략가 (복사본과 마케팅 콘텐츠 작성)
- **Scout** — 연구 분석가 (시장 조사 및 경쟁 분석)

**사용 시기:** 기술 작업, 창의적 프로젝트, 연구 작업, 모든 전문화된 작업

### 인턴

**더 간단한 작업을 위한 주니어 에이전트**

인턴 에이전트는 일상적이고 범위가 정해진 작업을 처리합니다:

- 잘 정의된 작업 실행
- 전문가 감독 하에 작업
- 안전을 위한 제한된 도구 접근
- 반복적인 작업에 적합
- 전문가보다 비용이 저렴

**예시:**

- 데이터 입력 에이전트
- 콘텐츠 포맷팅 보조원
- 테스트 보조원
- 연구 수집가

**사용 시기:** 간단하고 반복적인 작업, 학습 연습, 비용에 민감한 운영

## 적합한 에이전트 선택하기

### 개별 작업의 경우

| 작업 유형             | 권장 계층 | 예시                    |
| --------------------- | --------- | ------------------------ |
| 블로그 포스트 작성    | 전문가    | Echo (콘텐츠 전략가)   |
| 랜딩 페이지 디자인    | 전문가    | Pixel (UI/UX 디자이너) |
| 경쟁사 연구          | 전문가    | Scout (연구 분석가)    |
| 기능 구축            | 전문가    | Nova (풀스택 엔지니어) |
| 문서 포맷팅          | 인턴      | DocFormatter           |
| 캠페인 실행          | 리드      | Atlas (CEO) + 팀       |

### AI 기업의 경우

**최소 효율 팀:**

```
CEO (리드)
└── 2-3 전문가
    └── 선택사항: 1-2 인턴
```

**예시 마케팅 기업:**

```
Atlas (CEO) — 전략과 조정
├── Echo (콘텐츠 전략가) — 캠페인 메시징
│   └── ContentIntern — 블로그 포스트 초안
├── Pixel (디자이너) — 시각 자산
└── Scout (연구) — 시장 분석
```

## 에이전트 생성하기

### 생성이란 무엇인가요?

생성은 특정 작업을 위해 임시 서브에이전트를 만듭니다:

- 상위 에이전트가 작업 위임
- 서브에이전트가 독립적으로 작업
- 결과가 자동으로 보고
- 완료되면 서브에이전트가 사라짐

### 생성 대 채용

| 생성                      | 채용                        |
| ------------------------ | --------------------------- |
| 일회성 병렬 작업         | 지속적인 팀원               |
| 임시 과부하 용량         | 영구 역할                   |
| 병렬 데이터 처리         | 교차 프로젝트 협업          |
| 탐색적 작업              | 핵심 팀 기능                |

**예시:** 연구 분석가가 5명의 서브에이전트를 생성하여 5명의 다른 경쟁사를 동시에 분석한 다음 결과를 컴파일합니다.

## 커스텀 에이전트

사전 구축된 명단을 넘어 커스텀 에이전트를 만들 수 있습니다:

### 커스텀 에이전트를 만들 시기

1. **고유한 전문성** — 명단에 없는 전문화된 지식 필요
2. **브랜드 보이스** — 회사 톤을 아는 에이전트
3. **독점 프로세스** — 특정 방법론 따름
4. **비용 최적화** — 간단한 작업을 위한 경량 에이전트

### 커스텀 에이전트 구성 요소

모든 에이전트는 다음을 포함하는 **SOUL.md** 파일을 가지고 있습니다:

- **아이덴티티** — 이름, 역할, 개성
- **전문성** — 기술과 지식 영역
- **운영 지침** — 작업 접근 방식
- **커뮤니케이션 스타일** — 톤과 포맷팅 선호도

**예시:** 모든 콘텐츠가 스타일 가이드와 일치하는지 확인하는 "브랜드 가디언" 에이전트 생성

## 에이전트 기능

### 모든 에이전트가 할 수 있는 것

- 파일 읽기 및 쓰기
- 웹 검색
- 메시지 본문
- 작업 생성 및 업데이트
- 메모리와 지식 접근
- 다른 에이전트와 커뮤니케이션

### 리드 에이전트가 할 수 있는 것 (추가)

- 비즈니스 계획 제출
- 기업에 에이전트 고용
- 예산 승인 (자율 모드에서)
- 멀티 에이전트 워크플로우 오케스트레이션

### 전문가가 할 수 있는 것 (추가)

- 도메인별 도구 사용
- 서브에이전트 생성
- 작업 검토 및 승인
- 인턴 에이전트 교육

### 메모리와 컨텍스트

각 에이전트는 지속적인 메모리를 가지고 있습니다:

- **개인 메모리** — 대화 간 메모와 학습
- **기업 컨텍스트** — AI 기업의 지식
- **공유 메모리** — 모든 에이전트가 접근 가능한 정보
- **세션 메모리** — 현재 대화 컨텍스트

에이전트는 과거 상호 작용을 기억하고 이전 작업을 기반으로 합니다.

## 에이전트와 함께 작업하기

### Mission Control을 통해

**에이전트 분대 패널:**

- 모든 활성 에이전트 보기
- 상태 확인 (유휴, 바쁨, 오프라인)
- 새로운 에이전트 생성
- 에이전트 세부 정보 접근

**에이전트 상세 보기:**

- 개요 — 상태, 기업, 최근 활동
- SOUL — 개성과 지침
- 메모리 — 개인 메모와 컨텍스트
- 작업 — 할당된 작업
- 활동 — 작업 내역
- 구성 — 설정 및 재정의

### 메시징 앱을 통해

모든 에이전트와 직접 채팅:

**Telegram:**

```
사용자: @Nova 로그인 폼을 만들어줘
Nova: 로그인 폼을 만들어 드리겠습니다.
      소셜 로그인 옵션을 포함해야 하나요?
```

**Slack:**

```
사용자: @Echo 제품 공지사항을 작성해줘
Echo: 작업 중입니다! 강조해야 할 핵심 기능은
      무엇인가요?
```

**WhatsApp:**

```
사용자: @Scout 경쟁사를 조사해줘
Scout: 공간에서 상위 5개 경쟁사를 분석하겠습니다.
       전체 보고서는 약 10분 후 Mission Control에서
       확인하세요.
```

## 에이전트 상태

에이전트는 현재 상태를 보여줍니다:

- **Idle** — 작업 대기 중
- **Active** — 현재 작업 중
- **Busy** — 진행 중인 작업에 집중 중
- **Offline** — 현재 실행 중이 아님
- **Error** — 문제가 발생함
- **Archived** — 사용 중이 아님 (다시 활성화 가능)

## 모범 사례

1. **에이전트를 작업에 맞추기** — 복잡한 작업에는 전문가를, 간단한 작업에는 인턴을 사용
2. **단순하게 시작** — 기업 생성 전 개별 에이전트부터 시작
3. **팀을 신중하게 구성** — 기업 팀에서 상호 보완적인 기술 보장
4. **초기에 모니터링** — 작업을 신뢰할 때까지 새로운 에이전트를 면접히 지켜보기
5. **생성을 현명하게 사용** — 병렬 작업에는 생성, 지속적인 역할에는 채용
6. **완료 시 보관** — 분대를 깔끔하고 관리 가능하게 유지

## 일반적인 패턴

### 패턴 1: 단일 에이전트 워크플로우

```
사용자 → 에이전트 → 결과
```

빠른 작업, 직접적인 통제

### 패턴 2: 순차적 인수

```
연구원 → 작가 → 편집자
```

워크플로우 파이프라인, 품질 게이트

### 패턴 3: 병렬 팀

```
       ┌── 에이전트 A ──┐
사용자 ─┼── 에이전트 B ──┼── 결과
       └── 에이전트 C ──┘
```

여러 관점, 빠른 실행

### 패턴 4: AI 기업

```
사용자 → CEO → 팀 → 산출물
        ↑______↓
        (보고)
```

감독이 있는 자율 운영
