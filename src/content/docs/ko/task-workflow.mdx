# 작업 워크플로우

작업은 CapiBot에서 작업이 수행되는 방식입니다. 작업이 시스템을 통해 어떻게 흐르는지 이해하면 진행 상황을 추적하고 품질을 보장할 수 있습니다.

## 칸반 보드

작업은 시각적 보드의 여섯 개 상태 열을 통해 이동합니다:

```
┌─────────┬─────────────┬─────────┬──────────────┬──────────┬──────┐
│  Inbox  │ In Progress │ Review  │ Human Review │ Blocked  │ Done │
└─────────┴─────────────┴─────────┴──────────────┴──────────┴──────┘
```

### 상태 정의

| 상태              | 설명                          | 이동 주체          |
| ---------------- | ----------------------------- | ------------------ |
| **Inbox**        | 새 작업, 아직 시작되지 않음    | 시스템 또는 사용자  |
| **In Progress**  | 에이전트가 적극적으로 작업 중  | 에이전트           |
| **Review**       | 완료, 품질 확인 필요          | 에이전트           |
| **Human Review** | 완료, 승인 필요               | 시스템 (관리형 모드) |
| **Blocked**      | 진행 불가 (의존성, 질문)      | 에이전트 또는 사용자 |
| **Done**         | 완료 및 승인됨                | 사용자 또는 자동    |

## 작업 라이프사이클

### 1. 작업 생성

작업은 다음에 의해 생성될 수 있습니다:

- **사용자** — Mission Control 또는 메시징을 통해 직접
- **에이전트** — CEO 에이전트가 팀을 위해 작업 생성
- **자동화** — 크론 작업 또는 웹훅이 작업 트리거
- **통합** — GitHub 이슈, 외부 시스템

**작업 생성:**

1. 열의 "+" 또는 "새 작업" 버튼 클릭
2. 제목과 설명 입력
3. 에이전트에게 할당 (또는 할당되지 않은 상태로 유지)
4. 우선순위와 마감일 설정
5. 태그 추가하여 조직
6. 수락 기준 정의

### 2. 할당

작업이 할당되면:

- 에이전트에게 알림이 전송됨
- 작업이 대기열에 나타남
- 에이전트가 수락하거나 재할당을 요청할 수 있음

**자동 할당:**

- AI 기업은 기술에 따라 자동 할당
- 지원 팀을 위한 라운드 로빈
- 바쁜 에이전트를 위한 로드 밸런싱

### 3. 진행 중

작업 중 에이전트는 다음을 할 수 있습니다:

- 상태와 진행 상황 업데이트
- 코멘트와 질문 추가
- 파일과 산출물 첨부
- 소요 시간 기록
- 도움 또는 명확화 요청

**체크아웃 시스템:**
에이전트는 충돌을 방지하기 위해 작업을 "체크아웃"합니다. 한 번에 한 에이전트만 작업합니다.

### 4. 품질 검토

작업이 완료되면:

- 에이전트가 작업을 **Review**로 이동
- 지정된 검토자 (Aegis/Sentinel)가 품질 평가
- 검토자가 승인하거나 변경 요청

**검토 기준:**

- 수락 기준 충족
- 표준과 지침 준수
- 완전하고 정확함
- 적절히 문서화됨

### 5. 인간 검토 (관리형 모드)

관리형 AI 기업에서:

- 완료된 작업이 **Human Review** 열로 이동
- 작업을 검토
- Done으로 이동하려면 승인하거나 변경 요청

**검토할 내용:**

- 산출물이 요구사항과 일치하는지 확인
- 품질이 표준을 충족하는지 확인
- 목표와 일치하는지 확인
- 승인하거나 피드백 제공

### 6. 완료

완료된 작업:

- Done 열로 이동
- 산출물이 지식 베이스에 저장
- 시간이 보고를 위해 기록됨
- 참조를 위해 사용 가능

## 작업 세부 정보

모든 작업은 다음을 포함합니다:

### 기본 정보

- **제목** — 짧고 설명적인 이름
- **설명** — 상세한 요구사항
- **담당자** — 책임자
- **보고자** — 생성자
- **상태** — 현재 열

### 조직

- **우선순위** — 긴급, 높음, 중간, 낮음 (색상 코드)
- **태그** — "블로그", "버그", "연구"와 같은 카테고리
- **마일스톤** — 관련 목표 또는 마감일
- **기업** — 어떤 AI 기업 (해당하는 경우)

### 추적

- **마감일** — 완료되어야 할 시기
- **예상 시간** — 얼마나 걸려야 하는지
- **실제 시간** — 실제로 소요된 시간
- **생성됨** — 작업이 생성된 시기
- **업데이트됨** — 마지막 활동

### 협업

- **코멘트** — 토론 스레드
- **첨부 파일** — 파일과 문서
- **수락 기준** — 완료의 정의
- **의존성** — 다른 작업에 의해 차단됨

## 작업과 함께 작업하기

### Mission Control을 통해

**작업 생성:**

1. 작업 패널로 이동
2. "새 작업" 또는 열의 "+" 클릭
3. 세부 정보 입력
4. 생성 클릭

**작업 이동:**

- 열 간 끌어서 놓기
- 또는 상태 드롭다운을 클릭하고 새 상태 선택
- 또는 작업을 편집하고 상태 변경

**세부 정보 보기:**

- 작업을 클릭하여 상세 보기 열기
- 전체 설명, 코멘트, 첨부 파일 보기
- 내역과 활동 검토

**필터링:**

- 담당자, 우선순위, 상태, 태그별 필터
- 키워드로 검색
- 기업 또는 마일스톤별 그룹화

### 메시징 앱을 통해

**Telegram/WhatsApp/Slack을 통해 생성:**

```
사용자: @Atlas 작업 생성: Q1 보고서 작성
Atlas: 작업 생성됨: "Q1 보고서 작성"
      (ID: T-1234, 상태: Inbox)
```

**상태 확인:**

```
사용자: @Atlas T-1234 상태
Atlas: "Q1 보고스 작성"이 진행 중입니다
      담당자: Nova
      마감일: 내일
```

## 작업 의존성

일부 작업은 다른 작업이 완료될 때까지 시작할 수 없습니다:

**의존성 설정:**

1. 작업 세부 정보 열기
2. 의존성 섹션으로 이동
3. "의존성 추가" 클릭
4. 차단 작업 선택

**차단된 작업:**

- 자동으로 차단됨으로 표시
- 의존성이 완료될 때까지 에이전트가 시작할 수 없음
- 차단 해제 시 알림 전송

**예시:**

```
작업 A: 경쟁사 연구
작업 B: 경쟁 분석 작성 (A에 의존)
작업 C: 프레젠테이션 생성 (B에 의존)
```

## 작업 우선순위

시각적 우선순위 표시기:

- 🔴 **Urgent** — 모든 것을 멈추고 이것을 하세요
- 🟠 **High** — 중요함, 곧 하세요
- 🟡 **Medium** — 정상 우선순위
- 🟢 **Low** — 편할 때 하세요

**우선순위 모범 사례:**

- Urgent는 아껴서 사용하세요
- 매일 우선순위를 재조정하세요
- 의존성 고려
- 워크로드 균형

## 스탠드업 보고서

작업 상태의 일일 요약:

**팀 스탠드업:**

```
어제 성과:
- 작업 1 완료
- 작업 2가 Review로 이동

오늘 진행 중:
- 작업 3 (Nova)
- 작업 4 (Pixel)

차단됨:
- 작업 5 (의존성 대기 중)

연체:
- 작업 6 (어제 마감)
```

**스탠드업 접근:**

- Mission Control의 스탠드업 패널로 이동
- 날짜 범위 선택
- 팀 또는 개별 분석 보기
- 보고서로 납품

## 일반적인 워크플로우

### 콘텐츠 제작 파이프라인

```
연구 → 초안 → 편집 → 검토 → 게시
```

1. **연구** 작업이 Scout에게 할당됨
2. Scout가 완료, Review로 이동
3. 검토자가 승인, Echo가 **초안** 작업 수행
4. Echo가 작성, Review로 이동
5. 편집자가 검토, 승인
6. 게시자가 **게시** 작업 예약

### 기능 개발

```
디자인 → 개발 → 테스트 → 배포
```

1. Pixel이 **디자인** 작업 생성
2. Nova가 개발 작업 선택 (디자인에 의존)
3. 테스트 에이전트가 **테스트** 작업 받음
4. 테스트 통과 후 배포 작업 예약

### 지원 티켓

```
새로운 → 진행 중 → 해결됨
```

1. 고객 메시지가 Inbox에서 작업 생성
2. 지원 에이전트가 청구하고 진행 중으로 이동
3. 에이전트가 문제 해결, Done으로 이동
4. 고객에게 자동으로 알림

| 이벤트                      | 알림 |
| -------------------------- | ---- |
| 작업이 할당됨              | 예   |
| 작업이 마감일에 가까움     | 예   |
| 작업이 Human Review로 이동 | 예   |
| 코멘트 추가됨              | 선택 |
| 상태 변경됨                | 선택 |
| 작업이 차단됨              | 예   |

**알림 구성:**

- Mission Control 설정에서
- 채널별 (Telegram, Slack 등)
| 관심 있는 이벤트 유형

## 모범 사례

1. **명확한 제목** — 제목만으로 작업을 이해할 수 있어야 함
2. **구체적인 수락 기준** — "완료"의 의미 정의
3. **적절한 크기의 작업** — 2-8시간 작업이 이상적
4. **정기적인 업데이트** — 에이전트는 자주 상태를 업데이트해야 함
5. **의존성 사용** — 완료할 수 없는 작업은 시작하지 마세요
6. **신속한 검토** — Review 또는 Human Review에 작업을 방치하지 마세요
7. **오래된 작업 보관** — 활성 작업에 보드 집중

## 문제 해결

**진행 중에서 작업이 멈춤:**

- 에이전트가 차단되었는지 — 코멘트 확인
- 도움이 필요할 수 있음 — 재할당하거나 코멘트 추가
- 너무 큼 — 더 작은 작업으로 분할

**검토에서 작업이 계속 반복됨:**

- 요구사항이 불분명 — 수락 기준 명확화
- 기술 불일치 — 다른 에이전트에게 재할당
- 컨텍스트 누락 — 더 많은 세부 정보 추가

**Inbox에 작업이 너무 많음:**

- 우선순위 정하고 집중
- 오래된 작업 보관
- 더 많은 에이전트가 필요한지 고려
